## 1. 框架总览

### 1.1 目标

为 `restock_system` 制定一份**云交付就绪度评分卡（Delivery Readiness Scorecard）**，用于：

- 评判当前项目相对"上线到公网云服务器、A+B+C 综合场景"的整体进度
- 定位需要补强的具体环节，输出按优先级排序的 P0/P1/P2 行动清单
- 作为后续每次迭代后的"再评估基线"

### 1.2 评分对象与范围边界

**评分对象**：当前 git 仓库（`E:\Ai_project\restock_system`）中的**代码与配置文件**。

**包含**：
- `backend/`、`frontend/`、`deploy/`、`docs/`
- 数据库迁移、Dockerfile、docker-compose、Caddyfile、部署脚本
- 与外部依赖（赛狐 API）的交互逻辑

**不包含**（重要范围声明）：
- 开发工具链（`.specify/`、`.claude/`、个人 IDE 配置）
- 临时脚本与历史已弃用代码
- **服务器环境实地审计**——本项目当前仍在本地开发设备阶段，**没有云服务器实体**。本评分卡只判断"代码与配置对未来云交付的就绪程度"，不评估：
  - 服务器规格、操作系统版本、Docker 版本兼容性
  - 网络拓扑、防火墙规则、DNS 解析
  - 域名注册、TLS 证书签发可达性
  - 时区、宿主机环境变量
  - 任何需要登录到云服务器才能验证的内容

### 1.3 评分结构（三轴）

```
                        ┌──────────────────────┐
                        │   评分矩阵 8 × 9     │

                        └──────────────────────┘

                ▼                 ▼                 ▼
       8 模块（行）          9 维度（列）       N/A 标记
       M1 赛狐集成           D1 功能完整性     某些组合无意义
       M2 补货引擎           D2 代码质量
       M3 建议单与推送       D3 安全性 ⚠️
       M4 任务队列           D4 可部署性
       M5 前端数据页         D5 可观测性

       M7 基础设施           D7 可维护性
       M8 部署与交付         D8 性能与容量

                          ↓ 加权折算 ↓

                  4 个汇总组（高层视图）
                  G1 能不能上线
                  G2 上线后能不能稳
                  G3 坏了能不能救
                  G4 用着顺不顺
```

### 1.4 计分制度

- **每格分数**：5 级 Rubric（0/1/2/3/4 → 缺失/初级/可用/良好/优秀）
- **N/A 格**：不参与该模块/维度的平均分计算
- **模块得分**：该模块所有非 N/A 格的算术平均
- **维度得分**：该维度所有非 N/A 格的算术平均
- **汇总组得分**：组内涉及的维度按相关度（1/2/3）加权平均
- **总体得分**：4 个汇总组的算术平均，并配合硬约束（见 4.3）

### 1.5 执行流程（方案 B：按模块分批推进）

```
Phase 0：准备
   ├─ spec 写入并 commit（本文档）
   └─ writing-plans 产出执行计划

Phase 1：模块逐项审计（8 个 checkpoint）
   ├─ M1 → 用户确认 → 沉淀标尺一致性记录
   ├─ M2 → 用户确认
   ├─ ... ...
   └─ M8 → 用户确认

Phase 2：聚合与报告
   ├─ 维度交叉视图
   ├─ 4 汇总组得分
   ├─ 总体得分 + 交付门槛判定
   ├─ P0/P1/P2 补强清单
   └─ 最终报告 commit
```

**默认模块顺序**：M1 → M2 → M3 → M4 → M5 → M6 → M7 → M8

**总投入估算**：约 10-12 个对话回合（8 个模块 checkpoint + 1-2 个聚合回合 + 1 个 writing-plans 回合）。

### 1.6 角色分工

- **Claude**：阅读代码、跑命令（pytest / vue-tsc / grep）、对照 Rubric 给出每格的初评 + 证据引用（文件:行号）+ 疑点标注
- **用户**：对每个模块的初评结果做确认 / 反驳 / 调整；提供 Claude 无法从代码看出的业务上下文

## 2. 9 维度的 5 级 Rubric 判据

每条针对 **公网云部署 + 小团队** 场景定制；级别是**累积式**（3 = 2 的全部 + 额外项）。

### D1 功能完整性

| 级 | 判据 |
|---|---|
| 0 | 核心功能未实现；主链路存在断点 |
| 1 | 单个步骤可跑通，但端到端存在阻塞 |
| 2 | 主链路端到端跑通；常见场景覆盖；边界场景（空数据/超时/并发）行为不明 |
| 3 | 2 + 边界场景已处理；少量低频场景未覆盖但已记录 TODO |
| 4 | 3 + 异常恢复路径覆盖；契约/集成测试守护回归 |

### D2 代码质量

| 级 | 判据 |
|---|---|
| 0 | 无 lint/type check/test；存在巨型函数或大量重复 |
| 1 | 有部分测试；少量重复代码；无统一格式化 |
| 2 | lint + format 通过；核心模块单测 >50%；重复代码已识别 |
| 3 | 2 + type check 通过 + 单测覆盖 >70% + 命名清晰无明显代码异味 |
| 4 | 3 + 集成测试齐全 + mutation/property test + 0 ruff/mypy warning |

### D3 安全性 ⚠️

| 级 | 判据 |
|---|---|
| 0 | 无认证 / 明文密码 / SQL 注入风险 / 密钥硬编码 |
| 1 | 有 JWT 但密钥从配置文件读 / 输入校验不全 |
| 2 | JWT + 密钥走环境变量 + Pydantic 全量校验 + 密码 hash + 登录锁定 |
| 3 | 2 + 速率限制（IP/用户）+ CORS/CSRF 正确 + 依赖 CVE 扫描 + 日志脱敏 + TLS + 安全 headers（HSTS/CSP/X-Frame-Options） |
| 4 | 3 + 密钥轮换 + WAF/fail2ban + 渗透测试 + 完整审计日志 + 备份加密 + 最小权限 DB 账号 |

### D4 可部署性

| 级 | 判据 |
|---|---|
| 0 | 无 Dockerfile；靠开发者口传 |
| 1 | 有 Dockerfile 但需手动多步；环境变量未文档化 |
| 2 | docker-compose 一键启动 + .env.example + 迁移脚本 + 健康检查 |
| 3 | 2 + 一键部署脚本（备份/迁移/部署/回滚/smoke）+ 启动时配置校验 + 资源限制 + .env 与代码自动一致性校验 |
| 4 | 3 + 蓝绿/滚动部署 + IaC + 多环境（dev/staging/prod）+ CI/CD 自动部署 + 部署后自动 smoke test |

### D5 可观测性

| 级 | 判据 |
|---|---|
| 0 | 只有 print / 基础 logging；无 request_id |
| 1 | 有日志库；格式混乱；无 request 追踪 |
| 2 | 结构化日志（JSON）+ request_id 自动绑定 + 健康检查端点 |
| 3 | 2 + 业务事件指标（计算耗时/同步条数/推送成功率）+ /metrics + 日志统一收集 + 错误告警接入（飞书/邮件） |
| 4 | 3 + OpenTelemetry 追踪 + Grafana 看板 + SLO/SLI + 外部主动探测 |

### D6 可靠性

| 级 | 判据 |
|---|---|
| 0 | 无错误处理；外部 API 失败即崩；状态可能损坏 |
| 1 | 有 try/catch 但仅吞异常；无重试策略 |
| 2 | 关键外部调用有 retry + 超时；事务边界清晰；幂等保护到位（dedupe_key 等） |
| 3 | 2 + 指数退避 + 熔断/降级 + 死信队列 + 数据一致性校验任务 + 异常分类映射（BusinessError） |
| 4 | 3 + chaos test + 灾难恢复演练 + 故障注入 + 自动故障转移 |

### D7 可维护性

| 级 | 判据 |
|---|---|
| 0 | 无文档；目录混乱；新人无法独立启动 |
| 1 | 有 README 但已过时；目录约定靠口传 |
| 2 | AGENTS.md/CLAUDE.md 完整 + onboarding 可让新人独立启动 + 架构蓝图存在 |
| 3 | 2 + ADR 记录关键决策 + 模块边界清晰 + 文档同步协议执行 + runbook 覆盖常见故障 + 注释覆盖非显然逻辑 |
| 4 | 3 + 自动化文档生成 + 完整 ADR 历史 + onboarding < 1h + 架构图自动同步 |

### D8 性能与容量

| 级 | 判据 |
|---|---|
| 0 | 未知响应时间；存在 N+1；无优化 |
| 1 | 基础索引存在；未做性能测试；慢查询无人处理 |
| 2 | 关键接口 P95 < 1s；索引覆盖主要查询；无明显 N+1；前端首屏 < 3s |
| 3 | 2 + 接口 SLO 明确 + 慢查询日志 + 资源限制 + 前端分包 + 缓存策略 + 容量评估文档 |
| 4 | 3 + 负载测试基线 + 自动伸缩 + 数据库分区/读写分离 + CDN |

**注**：D8 等级 2 的"P95 < 1s"按实际情况校准；如该模块 SLA 要求更严或更松，校准记录写入 `_calibration.md`。

### D9 用户体验

| 级 | 判据 |
|---|---|
| 0 | 界面风格不统一；错误英文；操作无反馈 |
| 1 | 基础页面可用；风格不统一；错误提示笼统 |
| 2 | 统一组件容器（PageSectionCard）+ 中文界面 + 加载/错误反馈 + 表单基本校验 |
| 3 | 2 + 设计系统对齐 + 状态变化反馈 + 错误提示具体可操作 + 跨页选择/筛选体验良好 + 移动端基本可用 |
| 4 | 3 + 完整 a11y + 键盘快捷键 + i18n + 用户行为分析 + A/B 测试 |

### 关于 4 级"优秀"

**4 级 ≠ 必须达到的目标**。它代表"行业顶级实践"，对 1-5 用户场景大多数维度是过度工程化的。

**真实交付目标**：大多数维度 ≥ 3，关键维度（**D3 安全 / D4 部署 / D6 可靠性**）必须 ≥ 3，其他维度可接受 2。

## 4. 4 汇总组权重映射

### 4.1 权重表

| 维度 | G1 能不能上线 | G2 上线后能不能稳 | G3 坏了能不能救 | G4 用着顺不顺 |
|---|:--:|:--:|:--:|:--:|
| D1 功能完整性 | **3** | — | — | 1 |
| D2 代码质量 | 1 | 1 | — | 2 |
| D3 安全性 ⚠️ | **2** | **2** | — | — |
| D4 可部署性 | **3** | — | **2** | — |
| D5 可观测性 | — | **2** | **3** | — |
| D6 可靠性 | — | **3** | **2** | — |
| D7 可维护性 | — | — | 1 | 2 |
| D8 性能与容量 | — | **2** | — | 1 |
| D9 用户体验 | — | — | — | **3** |
| **权重合计** | **9** | **10** | **8** | **9** |

`—` = 该维度不参与该汇总组。

### 4.2 计算公式

每个汇总组的得分 = `Σ (该维度得分 × 该维度在该组的权重) / Σ (该维度在该组的权重)`

注意：每个维度的"分数"本身已经是该维度跨 8 模块的算术平均（N/A 不参与）。所以汇总组得分是**两层聚合**：

1. **维度内**：跨模块算术平均 → 维度得分（0-4）
2. **组内**：跨维度按权重加权平均 → 汇总组得分（0-4）

**示例**（假想数据）：

```
D1=3, D2=2, D3=2, D4=2

G1 = (3×3 + 2×1 + 2×2 + 2×3) / (3+1+2+3)
   = (9 + 2 + 4 + 6) / 9
   = 21 / 9
   = 2.33  →  接近"可用"
```

### 4.3 总体得分与交付门槛

```
总体得分 = (G1 + G2 + G3 + G4) / 4

交付门槛：
  ✅ 达标：总体 ≥ 2.5  且  4 组中没有任何一组 < 2.0  且  D3/D4/D6 维度均 ≥ 3.0
  ⚠️ 待补强：总体 ≥ 2.0 但不满足上述任一条件
  ❌ 不可上线：总体 < 2.0 或 任一汇总组 < 1.5 或 D3 < 2
```

### 4.4 设计意图说明

- **G1（能不能上线）**：D1 功能完整性和 D4 可部署性是 3 分核心；D3 安全 2 分作为基线；D2 代码质量 1 分（间接影响）
- **G2（能不能稳）**：D6 可靠性 3 分核心；D5 可观测、D8 性能、D3 安全各 2 分；D2 代码质量 1 分
- **G3（坏了能救）**：D5 可观测性 3 分核心——坏了第一件事是**看见**它；D6 可靠性、D4 可部署性各 2 分；D7 可维护性 1 分
- **G4（用着顺）**：D9 用户体验 3 分核心；D7 可维护性、D2 代码质量各 2 分
